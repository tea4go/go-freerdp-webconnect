# go-freerdp-webconnect Wails 桌面化改造建议

## 目标

将当前仓库从“本地启动一个 HTTP/WebSocket 服务，再由浏览器访问页面”的模式，改造成“用户直接启动桌面应用”的模式。

这里的“桌面化”建议采用 [Wails](https://wails.io/) 作为应用壳层：

- Go 继续承载 FreeRDP 和连接控制逻辑
- 前端运行在桌面应用内嵌 WebView 中
- 用户不再手动访问 `http://localhost:...`
- 后续可以逐步去掉旧的网页入口和遗留浏览器兼容逻辑

需要明确的是：Wails 并不是原生控件框架，它仍然是 HTML/CSS/JS 前端 + 桌面壳。因此它适合解决“不要网页版入口、要桌面应用分发和生命周期管理”的问题，但不会自动消除 Web 前端本身。

---

## 当前仓库现状

### 1. 当前应用本质上是本地 Web 服务

`main.go` 负责：

- 解析命令行参数
- 启动 `/ws` WebSocket 接口
- 启动 `/api/version`、`/api/config` HTTP 接口
- 将 `webroot` 目录作为静态站点发布
- 打印本地访问地址，要求用户打开浏览器访问页面

换句话说，当前程序不是桌面 App，而是“本地 Web 网关”。

### 2. RDP 后端和网页前端之间已经有清晰边界

`initSocket()` 的职责很单一：

- 从 WebSocket 请求里提取主机、用户名、密码、分辨率
- 创建 `sendq` / `recvq` / `inputq`
- 启动 `rdpconnect()`
- 将浏览器输入事件转成 `inputEvent`

这说明核心 RDP 能力并没有直接绑死在 HTTP 服务器上，而是绑在通道模型上。  
这是后续迁移到 Wails 的关键有利条件。

### 3. `rdp.go` 是真正的核心资产

`rdp.go` 已经完成了最难替换的部分：

- 通过 CGO 调用 FreeRDP
- 注册绘图回调
- 将 RDP 图形更新编码为二进制帧
- 将输入事件注入远端会话
- 支持动态分辨率调整

这部分不建议重写，建议尽量保留。

### 4. 前端是较老的浏览器实现

当前 UI 由 `webroot/index-debug.html` 和 `webroot/js/wsgate-debug.js` 驱动，特点是：

- 依赖 MooTools / simpletabs / 老式 DOM 事件模型
- 使用 Canvas 渲染 RDP 画面
- 使用原生 WebSocket 接收二进制图形数据
- 将鼠标、键盘、触摸事件编码后发回服务端
- 包含大量历史浏览器兼容逻辑

这套前端短期可复用，但不适合作为长期桌面产品形态。

## 改造判断

## 结论

这个仓库适合改成 Wails 桌面应用，且建议采用“保留 RDP 核心、替换宿主层、逐步收敛前端”的方式推进。

不建议一开始就推倒重来。  
最合理的做法是先把应用从“浏览器访问本地网页”切换到“Wails 桌面窗口”，再逐步去掉旧网页时代遗留的启动方式和前端框架。

---

## 推荐改造路线

## Phase 1: 先变成桌面应用，不先大改协议

目标是尽快把产品形态从网页变成桌面应用，同时控制风险。

建议：

- 新建 Wails 应用入口
- 保留现有 Go + FreeRDP 核心
- 前端先继续使用 Canvas 渲染思路
- 后端和前端之间仍保留本地二进制流通道
- 用户只启动桌面 App，不再手动打开浏览器

这一阶段有两种实现方式。

### 方案 A：Wails 前端仍连接本地 WebSocket

做法：

- Wails 启动后端时同时启动一个仅监听 `127.0.0.1` 的本地 WebSocket 服务
- Wails 窗口内的前端继续走现有 `wsgate` 二进制协议
- 前端静态资源由 Wails 内置而不是由浏览器访问外部地址

优点：

- 改动最小
- `rdp.go` 和二进制协议几乎不用动
- `wsgate-debug.js` 能较大比例复用
- 可以先验证 Wails 窗口中渲染性能和输入行为

缺点：

- 仍保留了一个本地 socket 服务
- 架构上不够“纯 Wails”
- 旧前端框架和旧协议仍然存在

### 方案 B：改为 Wails 事件/绑定传输

做法：

- 去掉 `/ws`
- Go 侧通过 Wails runtime event 或绑定接口把帧发给前端
- 前端通过 Wails API 把输入事件发回 Go

优点：

- 架构更贴近桌面应用
- 不再需要本地 HTTP/WebSocket 网关

缺点：

- 二进制高频帧流要重新设计传输方式
- 序列化/桥接成本更高
- 一开始就做，风险明显高于方案 A

### Phase 1 建议

优先选 **方案 A**。  
先把“浏览器壳”换成“桌面壳”，而不是同时重构宿主、协议、前端框架。

---

## Phase 2: 清理 Web 服务入口

当 Wails 桌面窗口稳定后，再逐步去掉当前为浏览器服务的 HTTP 形态。

建议拆分：

- 将 `main.go` 的 HTTP 路由注册逻辑抽离
- 将连接参数、版本信息、运行状态改成 Go service 层
- 让 Wails 直接调用 Go 方法获取配置和版本
- 去掉 `/api/version` 和 `/api/config`

这一阶段的目标是：

- 后端成为“RDP service”，而不是“HTTP server”
- Wails 成为正式入口
- 旧网页入口变成兼容层，最终删除

---

## Phase 3: 重写桌面端前端 UI

这一阶段才建议真正处理老前端问题。

### 建议保留的部分

- Canvas 渲染模型
- RDP 二进制消息解码逻辑
- 输入事件编码逻辑

### 建议替换的部分

- `index-debug.html`
- MooTools 事件系统
- `simpletabs`
- 旧式全局变量组织方式
- 历史浏览器兼容代码

### 建议的新形态

可以考虑使用：

- 原生 JS + 模块化
- 或 React/Vue/Svelte 任一现代前端方案

建议至少把 UI 层和 RDP 协议层拆开：

- `ui/` 负责连接页、设置页、错误提示、会话管理
- `rdp-client/` 负责 Canvas 渲染、帧协议解析、输入事件编码

这样后续维护成本会明显低很多。

---

## Phase 4: 做成真正可分发的桌面产品

Wails 真正带来的价值不只是窗口壳，还包括桌面产品能力。

后续可以逐步加上：

- 本地连接历史
- 凭据安全存储
- 多会话窗口或标签页
- 应用菜单
- 系统托盘
- 自动更新
- 安装包分发

这部分属于产品化，而不是仅仅“把网页装进窗口”。

---

## 建议的代码重组

建议后续目录朝这个方向演进：

```text
.
├── cmd/
│   ├── desktop/           # Wails 桌面入口
│   └── web-legacy/        # 可选，保留旧网页模式一段时间
├── internal/
│   ├── rdp/
│   │   ├── session.go     # 会话生命周期
│   │   ├── freerdp.go     # CGO / FreeRDP 封装
│   │   ├── protocol.go    # 帧与输入事件定义
│   │   └── service.go     # 给桌面壳调用的 service
│   ├── desktop/
│   │   └── app.go         # Wails 绑定对象
│   └── web/
│       └── legacy.go      # 旧 HTTP/WebSocket 入口
├── frontend/              # Wails 前端
├── webroot/               # 临时保留，逐步淘汰
└── docs/
```

这不是第一步就要一次做完的事情，但建议尽早按这个方向切。

## 必须处理的问题

## 1. 密码传递方式不适合桌面应用

当前模式里：

- 前端把用户名、密码放到 WebSocket query string
- `/api/config` 还会把密码直接返回给前端

这在桌面产品里不合适。

建议改成：

- 前端通过 Wails 绑定直接传结构化参数
- 凭据不出现在 URL 中
- 后续接操作系统安全存储

---

## 2. 构建和打包难点不在 UI，而在 FreeRDP 动态库

当前 macOS 仍依赖手动设置：

- `DYLD_LIBRARY_PATH`

Linux / Windows 也都依赖特定方式打包 `libfreerdp`。

这意味着后续真正的桌面分发难点是：

- 怎么把 FreeRDP 相关动态库和应用一起打包
- 怎么保证 Wails 产物在目标机器可直接运行

因此建议尽早验证：

- macOS Wails 打包后的 dylib 装载路径
- Windows 安装包中的 DLL 布局
- Linux AppImage / deb / rpm 的依赖策略

---

## 3. 当前测试体系绑定在 HTTP/WebSocket 模型上

现有测试主要验证：

- HTTP 服务是否可启动
- `/api/version` 是否可访问
- WebSocket 握手是否成功

迁移后需要把测试重新分层：

- RDP core 测试
- 桌面 service 测试
- 前端渲染 smoke test
- 可选保留 legacy web 测试

否则一旦删掉 HTTP 入口，现有测试基本都会失效。

---

## 4. 多会话能力目前基础较弱

当前 `rdp.go` 中动态分辨率通道依赖全局变量 `g_dispCtx`。  
这更像单连接设计。

如果未来要支持：

- 多窗口
- 多标签页
- 多远程会话并发

那么需要把这类全局状态改造成 per-session 结构。

---

## 不建议的改法

### 1. 不建议先重写 FreeRDP 接入层

原因：

- 这是当前仓库最复杂、最稳定的资产
- 风险高、回报低
- 与“改成桌面端”目标并不直接相关

### 2. 不建议一开始就彻底删除 `webroot`

原因：

- 现有 Canvas 协议实现还可以复用
- 先保证桌面壳跑通，再做前端重构更稳

### 3. 不建议第一阶段就强行把所有帧流改成 JSON/RPC

原因：

- 这是高频图形更新场景
- 当前二进制协议更适合这个用途
- 过早引入序列化重构会明显放大风险

---

## 最小可行改造方案

如果目标是“尽快看到桌面版跑起来”，建议做一个 MVP：

### 后端

- 新增 Wails 入口
- 抽取当前连接参数结构
- 保留 `rdpconnect()` 和现有帧通道模型
- 暂时保留本地 WebSocket 桥接

### 前端

- 新建 Wails 前端壳
- 首版直接承载现有 Canvas 客户端能力
- 简化连接表单
- 去掉老旧浏览器探测逻辑

### 产品形态

- 启动应用即打开桌面窗口
- 连接配置在窗口内输入
- 连接建立后在同一窗口显示远程桌面
- 不再向用户暴露浏览器访问入口

这样可以在较短周期内完成“从网页版到桌面端”的第一步。

---

## 推荐实施顺序

1. 新建 Wails skeleton，不动 `rdp.go`
2. 把 `main.go` 中非 HTTP 的部分抽成 service
3. 先保留本地 WebSocket 桥接，让 Wails 前端能显示 RDP 画面
4. 清理掉命令行默认凭据、URL 传密码、`/api/config` 这类不合适设计
5. 重写连接页面和桌面端 UI
6. 重新设计测试
7. 处理各平台打包和动态库分发
8. 最后删除 legacy web 模式

---

## 最终建议

这次改造应当被视为“宿主层替换 + 产品形态升级”，而不是“重写 RDP 客户端”。

最推荐的策略是：

- 保留 FreeRDP 与 Go 通道模型
- 第一阶段仅替换为 Wails 桌面壳
- 第二阶段再去网页化残留
- 第三阶段再做现代化前端和产品化能力

如果直接一步做成“完全无 HTTP、完全重写前端、完全重构协议”的版本，失败概率会显著上升。

相反，采用分阶段迁移，可以在较低风险下快速得到一个可运行的桌面版，并且给后续重构保留足够余地。
